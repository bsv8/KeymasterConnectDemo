# KeymasterConnectDemo 三段式布局与三栏工作台硬切换施工单

## 1. 结论先行

本次直接做**硬切换**，不分步骤实施，不保留旧 tab 顶栏布局，不做双布局共存。

原因很直接：

- 当前页面已经不是 2 到 3 个功能的小 demo，而是 7 个协议方法 + 1 个工具区的协议工作台。
- 继续沿用“上方一排 tab”只会让方法继续横向膨胀，信息密度失控，桌面端和移动端都会越来越别扭。
- 这个项目的定位是**协议验证操作台**，不是产品展示页；它更需要稳定的导航、稳定的主工作区、稳定的观察区，而不是继续给 tab 挤空间。
- 如果这次做成“先补一点左侧菜单，再暂时保留旧 tab”，最终只会多一套并存结构，增加样式和状态复杂度，没有真实价值。

所以这次要一次性把页面结构从：

- 顶部 hero / 配置
- 中部 tab 条
- 下部当前 tab + 右侧日志

切到：

- 上部 `header`
- 中部 `mainbody`
- 下部三栏工作区

这是页面结构级硬切换，不是局部补丁。

## 2. 这次到底要改成什么

改完后，页面结构固定为三段：

### 2.1 顶部 `header`

职责：

- 显示页面名称与用途
- 显示 popup 连接状态
- 显示当前 `targetOrigin`
- 显示当前激活测试项

要求：

- 轻量、稳定、可扫读
- 不再保留当前偏展示型 hero 双栏
- 不在这里放大块说明文案

### 2.2 中部 `mainbody`

职责：

- 承载全局公共配置
- 承载跨方法共享的上下文摘要

建议内容：

- `Keymaster Target Origin`
- `Popup Width`
- `Popup Height`
- `Ready Timeout(ms)`
- `Result Timeout(ms)`
- 测试钱包摘要
- 最近一次 Keymaster 主网地址摘要

要求：

- 这里只放**全局公共信息**
- 不混入具体方法的表单
- 不把这里做成第二个 hero

### 2.3 下部三栏工作区

#### 左栏：测试项目菜单

职责：

- 替代旧 tab 顶栏
- 显示所有测试项入口
- 显示每项状态
- 允许用户切换当前工作项

范围：

- `identity.get`
- `intent.sign`
- `cipher.encrypt`
- `cipher.decrypt`
- `p2pkh.transfer`
- `feepool.prepare`
- `feepool.commit`
- `test wallet`

#### 中栏：主工作区

职责：

- 只显示当前激活测试项的完整表单与主操作
- 显示该测试项最关键的结果摘要

要求：

- 始终只有一个当前工作面
- 不同时展开多个完整方法页面
- 不做多标签并排编辑

#### 右栏：观察区

职责：

- 显示当前测试项的 request 预览
- 显示当前测试项的 response / inspection / 原始结果
- 显示全局协议日志

要求：

- 右栏是**观察区**，不是第二个完整测试页
- 右栏必须稳定可预期，不能和中栏重复一套完整表单
- 日志保持全局口径，不按方法分别存一套

## 3. 为什么只能这么做

### 3.1 为什么不用更多 tab

因为问题不是“tab 样式不够宽”，而是交互模型已经不适合继续增长。

方法数继续增加时：

- 桌面端会越来越挤
- 移动端会退化成水平滚动长条
- 用户切换方法时，导航、操作、观察都混在一起

这不是样式问题，是结构问题。

### 3.2 为什么右栏不能做成第二个完整测试页

如果右栏也做成完整测试页，会立刻出现下面的问题：

- 两边都可以改表单，状态边界不清楚
- 同一方法是否允许双编辑面，不清楚
- 同一时刻只允许一条在途 request，但界面却像支持并行工作
- 页面复杂度显著上升，但协议能力没有增加

这违反当前项目“简单粗暴、失败就暴露、不要为了完整性增加系统复杂度”的原则。

### 3.3 为什么中部要放全局配置，不放方法表单

因为这些配置是页面级公共真值：

- 它们影响 popup session client
- 它们不是某个单独方法的私有参数
- 它们变化时需要重置会话

把它们混在某个方法页面里，会误导用户把公共配置理解成局部参数。

## 4. 怎么做

## 4.1 页面结构直接切到三段式

`src/App.tsx` 顶层布局直接重排为：

- `app-shell`
- `app-header`
- `app-mainbody`
- `workbench-layout`

其中 `workbench-layout` 固定为三栏：

- 左：`workbench-nav`
- 中：`workbench-main`
- 右：`workbench-observer`

不保留旧的：

- `hero`
- `hero-copy`
- `hero-panel`
- `tab-stage`
- `tab-strip`
- `tab-button`

能删就删，不保留兼容壳。

## 4.2 导航模型从 tabItems 保留语义，改为菜单项

当前 `tabItems` 的数据语义可以保留，但视觉和 DOM 结构改成菜单。

要求：

- 继续用一份方法清单驱动导航
- 每个菜单项展示：
  - 方法名
  - 中文说明
  - 当前状态
- 当前激活项高亮

不做：

- 分组折叠菜单
- 拖拽排序
- 收藏常用项
- 菜单搜索

## 4.3 当前工作面仍然只有一个 `activeTab`

虽然视觉结构改了，但交互真值继续保持简单：

- 仍然只保留一个 `activeTab`
- 中栏只渲染 `renderActiveTab()`
- 不引入多活动页状态
- 不引入缓存多个展开工作面

如果后续方法再增加，先继续增加菜单项，不要预先发明工作区分页系统。

## 4.4 右栏观察区从“仅日志”升级为“当前方法观察 + 全局日志”

右栏拆成两个区块：

- 当前方法观察区
- 全局协议日志区

当前方法观察区应根据 `activeTab` 切换：

- `identity.get`：看 request / raw result / decoded envelope / claims
- `intent.sign`：看 request / raw result / decoded envelope / 验签摘要
- `cipher.encrypt` / `cipher.decrypt`：看 request / raw result / 二进制结果
- `p2pkh.transfer`：看 request / raw result / tx 结果摘要
- `feepool.prepare` / `feepool.commit`：看 request / raw result / 关键回填字段
- `test wallet`：看钱包摘要 / UTXO / 回款结果

要求：

- 右栏和中栏信息分工明确
- 中栏偏输入与主操作
- 右栏偏观察与复核

## 4.5 样式直接重做，不在旧 tab 样式上打补丁

`src/styles.css` 这次应直接切掉旧 tab 样式块，围绕新三段式结构重建。

桌面端建议比例：

- 左栏：`220px` 到 `260px`
- 中栏：`minmax(0, 1fr)`
- 右栏：`380px` 到 `460px`

风格要求：

- 保留项目现有偏浅色、半透明、轻玻璃感方向
- 但回收过强的展示性
- 更偏“操作台”而不是“宣传页”

不做：

- 大幅新视觉实验
- 华而不实动效
- 复杂吸顶联动

## 4.6 结果展示从“混在方法页”改成“主摘要在中栏，细节在右栏”

这次要收紧信息层级。

原则：

- 中栏保留当前方法最关键、最需要立即判断的结果
- 右栏保留原始报文、解码结果、细节复核

例如：

- `identity.get`
  - 中栏：本地验签是否通过、最近一次地址、关键 claims 摘要
  - 右栏：request / raw result / decoded envelope / resolvedClaims
- `feepool.prepare`
  - 中栏：`operationId`、`action`、是否成功
  - 右栏：完整 `result`

不要把所有 `ResultPanel` 原封不动继续堆在中栏里。

## 4.7 设计文档同步改真值

`docs/KeymasterConnectDemo-首版设计.md` 需要同步更新页面结构描述，避免文档继续写旧结构。

至少要把“页面包含 7 块协议能力区 + 1 个工具区 + 1 个公共结果区 + 1 个公共日志区”更新为更贴近真实布局的描述：

- 顶部轻量 header
- 中部全局配置与共享上下文
- 下部三栏工作台

文档不需要写成视觉稿，但必须和实现真值一致。

## 5. 不能怎么做

本次明确禁止以下做法：

- 不能保留旧 tab 顶栏，再额外补一个左侧菜单。
- 不能让中栏和右栏都变成完整可编辑测试页。
- 不能为了“看起来信息全”把所有结果面板都继续堆在中栏。
- 不能把全局配置塞进某个具体方法页面。
- 不能因为移动端麻烦就简单隐藏左栏测试菜单。
- 不能引入新的全局状态管理库、路由库、UI 组件库。
- 不能为了适配三栏布局，重写协议逻辑、popup session client 或测试钱包核心逻辑。
- 不能新增“多方法并行运行”“多活动工作面”“请求排队器”。
- 不能把日志拆成每个方法单独维护一份历史真值。
- 不能为了兼容旧结构留下大量废弃样式和废弃 DOM 壳。

## 6. 特殊情况应该怎么办

### 6.1 窄屏怎么办

移动端和窄屏不做复杂抽屉方案，直接按最简单方式退化：

- 三栏改为纵向堆叠
- 顺序固定为：
  - 菜单
  - 主工作区
  - 观察区

不要做：

- 左栏抽屉
- 右栏悬浮 inspector
- 双层 tab

原因很简单：这是协议工作台，不是高频移动端产品。能稳定读写和观察，比花哨响应式更重要。

### 6.2 当前有请求在途时切换菜单怎么办

允许切换查看，但不允许制造并行执行。

具体要求：

- `anyBusy === true` 时，方法菜单仍可切换查看
- 但当前在途 request 对应的方法提交按钮保持忙态
- 其它方法提交按钮也保持禁用

这样做的原因：

- 用户可能需要边等待边切去看日志或别的方法结果
- 但不能误导用户以为可以并行发第二条 request

### 6.3 修改全局配置时怎么办

当下面任一字段变化时：

- `targetOrigin`
- `popupWidth`
- `popupHeight`
- `readyTimeoutMs`
- `resultTimeoutMs`

继续沿用当前逻辑：

- 主动关闭旧 session client
- 下次提交重新建新会话

但页面表单状态不应被清空。

原因：

- 会话句柄应该重置
- 用户输入不属于会话资源，不该一起丢

### 6.4 右栏当前方法还没有任何结果怎么办

显示空态，而不是复用上一个方法的旧结果。

空态应明确告诉用户：

- 当前方法还没有请求预览或返回结果
- 可以先在中栏填写并运行

不能做：

- 菜单切换后右栏继续显示上一个方法的细节

因为这会制造“看起来像当前方法数据，实际是别的方法数据”的歧义。

### 6.5 日志太长怎么办

继续保持当前“最近 60 条”的全局日志策略。

不做：

- 分方法日志
- 日志搜索
- 日志分页
- 导出日志

这次目标是把结构收稳，不是把日志系统做大。

### 6.6 工具区信息和协议区信息关系怎么处理

`test wallet` 继续作为一个独立菜单项存在，但它属于同一工作台，不是外挂区域。

要求：

- 左栏菜单和其它协议方法平级
- 中栏显示测试钱包与手动回款完整操作
- 右栏显示钱包摘要、UTXO、回款结果

不能再把工具区做成“特殊大杂烩角落”。

## 8. 最终验收清单

### 8.1 结构验收

- 页面顶部是轻量 `header`，不再是旧 hero 双栏。
- 页面中部是全局公共配置与共享上下文。
- 页面下部是左菜单、中工作区、右观察区三栏结构。
- 桌面端不再出现一整排横向测试 tab。

### 8.2 行为验收

- 左栏所有测试项都可切换。
- 中栏始终只展示一个当前测试项的完整操作面。
- 右栏会跟随当前测试项切换观察内容。
- 协议日志仍然是全局统一日志。
- 同一时刻仍然只允许一条在途 request。
- 修改全局配置后，会话重置，但用户已填表单不会被清空。

### 8.3 特殊情况验收

- 窄屏下页面会退化成“菜单 -> 主工作区 -> 观察区”的纵向结构。
- 当前方法无结果时，右栏显示空态，不串上一个方法数据。
- 忙态期间可以切换查看，但不能并行提交第二条 request。

### 8.4 代码边界验收

- 本次改动不引入新状态管理库。
- 本次改动不引入路由。
- 本次改动不新增并行请求机制。
- 本次改动不重写协议层或测试钱包核心逻辑。
- 本次改动不保留废弃旧 tab 结构作为兼容壳。

## 9. 实施提醒
